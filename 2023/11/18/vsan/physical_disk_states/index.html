[vSAN] 物理硬碟狀態

發表於 2023-11-18 | 分類於 vSAN | |  閱讀次數:
字數統計: 764

Disk由於晶圓問題、firmware問題、driver問題等等會造成ㄧ定機率上的故障，若一塊硬碟，他每天的故障機率是1%，一座資料中心假如有一萬個disk，代表每天有一百塊硬碟損壞，當然，vSAN對於部分類型損壞會嘗試自動修復，這樣能夠大幅減少許多人力成本，畢竟大部分的問題重開機就能解決。

LSOM是最底層與硬碟打交道的Component，他的DDH(Degraded Device Handling) Service可以幫忙監控硬碟狀態，當DDH偵測到Disk即將要故障或損壞事件，就把Disk標記為有問題的裝置，讓他不參與資料的I/O流程，更多細節，可以參考VMware KB: https://kb.vmware.com/s/article/2148358

這裡列舉vSAN偵測到的硬碟狀態以及他的原因

PDL

Permanent Disk Loss(PDL)，是一種嚴重的錯誤，需要人工介入，一般情況下是Host與Disk的連結出了問題，沒法連上去，ESXi Host內部或是Disk的bug都有可能

Transient Error

這種錯誤可以藉由I/O Retry幾次就可以修復，一般來說也是由於ESXi Host內部或是SCSI的bug，例如I/O的workload突然竄上來

Stuck I/O

Stuck I/O是發生在ESXi Host外面的，一般是由於firmware或hardware的bug導致I/O一直沒法完成，在vSAN 7.0 Update 3以前，發生了Stuck I/O，vSAN會把整個Host都關機，一個disk的I/O沒有反應會影響整個Cluster處理I/O的進程，不重啟是因為自動重啟之後，I/O還是會繼續Stuck，最好是手動的把Disk移除再重啟，vSAN 7.0 Update 3之後，會在監控平台提示使用者某個Disk發生了Stuck I/O，需要手動的重啟Host

URE

Unrecovered read error(URE)是LSOM遇到metadata read error所遇到的問題，這是由於Disk介質的錯誤，他能夠被自動修復。當Disk發生URE並且DDH打開時，vSAN會藉由把資料撤出並且重構disk來處理如此錯誤

High Latency

很好理解，當LSOM偵測到過多的High Latency I/O就會觸發

Congestion

當LSOM偵測到如下Congestion時，LSOM會通知DOM降低I/O flow直到Congestion解除

  1. Log Congestion
  2. SSD Congestion
  3. Component Congestion
  4. Memory Congestion
  5. IOPS Congestion

最後，當DDH檢測到Disk的錯誤後，就需要把Disk上Object的Component即時撤離，這時候就會需要用到Evacuation操作

Evacuate

由於Disk發生了錯誤，所以需要把Disk上的Component撤出來給其他Disk，以確保Object能服從使用者設定的Policy，當然也有可能Evacuate發生錯誤，這時就需要人工來介入了。關於Evacuation的結果，有如下三種

  1. Data evacuation completed
    成功了的意思
  2. Data evacuation failed due to insufficient resources
    顧名思義，這是由於Cluster的容量不足所觸發的錯誤，需要人工介入
  3. Data evacuation failed due to inaccessible objects
    當ESXi host重啟或是發生網路問題，就會容易觸發inaccessible object，這時候也需要人工介入看是否把Infra的問題修了，或是是否可以允許刪除該object
